# Phase 7 / Plan 02 — Multi-DEX Scale-Up + v1.0 Release

## Context

Plan 07-01 ships operability (config + metrics + ledger wiring +
per-cycle tip accounting). Once that lands, the engine runs for
multi-day sessions and emits structured metrics, but it still only
supports one DEX (Raydium AMM v4) and lacks the v1.0 release
artifacts (docs, reproducible-run script, versioned release).

This plan closes Phase 7 by:
1. Adding Orca Whirlpool and Meteora decoders (multi-DEX routing).
2. Adding a Prometheus / OpenTelemetry metrics adapter
   (07-01 ships the `MetricsSink` trait; this plan ships a
   production-grade backend).
3. Shipping the v1.0 release: documentation, reproducible-run
   script, versioned tag, changelog.

## Why this split

07-02 is the larger of the two Phase 7 plans because it touches:
- A new AMM family (Orca Whirlpool: concentrated liquidity,
  different math than constant-product).
- A new program (Meteora DLMM: bin-based liquidity, again
  different math).
- A new external dependency (Prometheus / OTel).
- A release process (tagging, signing, publishing).

All of these deserve their own review. Splitting them keeps 07-01
focused on the operability story and 07-02 focused on the
scale-up + release story.

## Goal

Make the engine v1.0:
1. Multi-DEX (Raydium + Orca + Meteora routing).
2. Production-observable (Prometheus scrape endpoint or OTel
   export).
3. Reproducible (a single script that replays a captured window
   and produces a ledger + recon report from scratch).
4. Documented (a `v1.0.0` release of the project, with a
   CHANGELOG, README, and a "How to reproduce this paper PnL
   report" walkthrough).

## Scope

In scope:
- `crates/dl-state/src/decoder/orca_whirlpool.rs` (new):
  Whirlpool account decoder (concentrated liquidity, ~250 B
  account layout, ticks + sqrt_price).
- `crates/dl-state/src/decoder/meteora_dlmm.rs` (new):
  Meteora DLMM account decoder (bin-based, ~1 KB account).
- `crates/dl-sim/src/fill_orca.rs` (new): concentrated-liquidity
  fill math (different from constant-product). No `sqrt`
  / `log2` — use a precomputed tick table or a polynomial
  approximation in fixed-point.
- `crates/dl-detect/src/graph.rs` (modify): per-DEX node labeling
  so multi-DEX cycles can be enumerated. Today the graph is
  token-keyed; add a `(dex_id, pool_id)` tag to edges.
- `crates/dl-app/src/metrics_prom.rs` (new): `MetricsSink` impl
  that exposes a `/metrics` HTTP endpoint (Prometheus format).
  Or, equivalently, an OTel exporter. Pick one for v1.0.
- `crates/dl-app/src/metrics_otel.rs` (new): same, OTel. The
  `MetricsSink` trait allows both; the user picks at startup.
- `docs/v1.0.md` (new): v1.0 release notes. What ships, what
  doesn't, how to reproduce a paper PnL report from a captured
  window.
- `scripts/reproduce_paper_pnl.sh` (new): single-command
  reproduction. Captures a 60s window if no input is given;
  otherwise replays an existing capture. Runs detect →
  simulate → ledger → recon. Writes a Markdown report.
- `CHANGELOG.md` (new): what changed since v0.x.
- `README.md` (existing, modify): update to v1.0 framing.

Out of scope (v1.1+):
- Sandwiching (per `solana-mev-landscape.md` §1.5, explicitly
  out of scope for the project).
- BAM-era tip routing (per 06-02 research doc §1.6 open question).
- Other DEXes (Phoenix, OpenBook, etc.).
- Live trade execution (v1.0 is paper-trading only).
- Real-time dashboard UI (a Grafana import of the Prometheus
  output is the v1.0 path; a custom UI is future).

## Research gate

**Medium — needs three sources:**

1. **Orca Whirlpool math.** The Whirlpool program uses
   concentrated liquidity with a `sqrt_price` Q64.64 fixed-point
   representation. Source: orca-so/whirlpools SDK on GitHub.
   No public PDF; the SDK source is the primary source. The plan
   must verify tick math against the SDK's `Position` math.

2. **Meteora DLMM math.** Bin-based liquidity with a custom
   bin-step. Source: meteora-ag/dlmm-sdk on GitHub. Similar
   situation to Orca — no PDF, SDK is the source.

3. **Prometheus / OTel crate choice.** `prometheus` (Rust) vs
   `opentelemetry` (Rust) vs writing a minimal text-format
   emitter. Plan must pick one and document the trade-off.

The plan is **blocked** until a research doc is committed at
`.paul/research/multi-dex-math.md` (or equivalent) covering all
three. The doc can be a methodology + source-link document; the
actual math derivation happens during implementation, with
specific function signatures cross-checked against the SDK
sources.

## Acceptance criteria

- **AC-1**: Orca Whirlpool decoder round-trips a real mainnet
  Whirlpool account. Verified by an integration test that
  fetches a known Whirlpool (via `dl-feed` if network is
  available, else from a captured fixture) and asserts the
  decoded `Pool` matches the SDK's reference output to within
  ±1 tick.

- **AC-2**: Meteora DLMM decoder round-trips a real mainnet
  Meteora DLMM account. Same shape as AC-1.

- **AC-3**: Orca fill math matches the SDK's reference
  `getAmountOut` for a known input/output pair to within ±1
  base unit (i.e., the fixed-point approximation's quantization
  error, not the math's error). Verified by a test with at
  least 3 input/output pairs.

- **AC-4**: Multi-DEX cycle detection works. A triangle
  involving Raydium + Orca + Meteora pools around a common
  token triplet is detected by `find_negative_cycles`.
  Verified by a synthetic fixture with one pool per DEX.

- **AC-5**: `dl-app metrics prom` (or `otel`) starts an HTTP
  server on a configurable port, exposes `/metrics` in
  Prometheus text format (or OTLP gRPC for OTel), and the
  engine's metrics stream live to it during a capture. Verified
  by a smoke test that scrapes `/metrics` and asserts at least
  5 distinct metric names are present.

- **AC-6**: `scripts/reproduce_paper_pnl.sh` works end-to-end.
  Given a capture file (or with no input, after a 60s live
  capture), it produces:
  - A `DLD-LDG1` ledger.
  - A `ReconReport` (printed + saved as JSON).
  - A Markdown summary at `out/PAPER_PNL_REPORT.md`.
  Verified by running the script against a fixture and asserting
  the three outputs exist and are non-empty.

- **AC-7**: `docs/v1.0.md` is committed, covers what ships,
  what doesn't, and how to reproduce. `CHANGELOG.md` is
  committed with entries for v1.0.0. `README.md` is updated to
  v1.0 framing.

- **AC-8**: `cargo build --workspace`, `cargo test --workspace`,
  `cargo fmt --all`, `cargo clippy --workspace --all-targets -- -D warnings`
  all clean. Test count ≥ 360 (was 320 after 07-01; +40 from
  AC-1, AC-2, AC-3, AC-4, AC-5, AC-6, AC-7 tests).

- **AC-9**: Float-free CI guards still pass. Orca + Meteora
  decoders are integer-only (the Whirlpool Q64.64 is u128
  arithmetic, no `f64`; the Meteora bin math is integer + small
  lookup tables). New guards: `dl-state/tests/fixed_point_no_floats.rs`
  covers the new decoders.

- **AC-10**: v1.0 git tag created (`git tag -s v1.0.0`).
  Reproducible build verified: `cargo build --release` produces
  bit-identical binaries on a second host (within ±1 byte on
  the strip step, which is acceptable for Cargo's deterministic
  build).

## Risks

- **Risk 1: Orca Whirlpool math is genuinely hard.** The
  concentrated-liquidity math involves a `sqrt` of a Q64.64
  number, which we either compute in fixed-point or precompute
  as a tick table. The plan assumes the latter (tick table) is
  acceptable; if accuracy demands the former, this becomes a
  significant fixed-point sqrt implementation. Mitigated by
  starting with the tick-table approach and documenting the
  accuracy tradeoff in `docs/v1.0.md`.

- **Risk 2: Meteora DLMM bin math.** Meteora's SDK is less
  mature than Orca's. The plan accepts the SDK as the source
  of truth and pins a specific SDK version in `Cargo.toml`.
  If the SDK changes its math, we have to update.

- **Risk 3: Prometheus / OTel choice.** Picking wrong costs
  rework. Mitigated by supporting both backends behind the
  `MetricsSink` trait; users pick at startup. The plan delivers
  both, but flags the choice as a v1.1 follow-up if the
  ecosystem shifts.

- **Risk 4: Release tagging friction.** A signed tag requires a
  GPG key, which may not be configured on the build host.
  Mitigated by allowing the tag to be unsigned initially and
  signed later from a configured host.
