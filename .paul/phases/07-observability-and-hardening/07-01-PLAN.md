# Phase 7 / Plan 01 — Metrics, Observability, Config-Driven Params

## Context

Phase 6 closed with a structurally-complete recon harness (dl-recon
+ dl-recon-overfit + dl-app recon CLI) but with a Phase 6 audit
flagging several honest deferrals. This plan absorbs three of them
plus the new scope from `.paul/ROADMAP.md`:

**Audit-flagged items this plan closes:**
1. `DL_LEDGER_PATH` wiring into `dl-app` (deferred from 05-02 AC-6,
   never re-deferred in writing). `run_dry_run` currently doesn't
   open a `LedgerWriter`. This is a v1.0-readiness gap.
2. Per-cycle tip accounting in `dl-sim` (06-02 honest deferral).
   Three of the six `engine_aggregate()` mappings in
   `dl-recon::onchain` return zero placeholders
   (`MeanTipLamports`, `TipAsPctOfMev`, `Median/P95WinnerPnlSol`).
   The harness compiles and produces the right *shape* of
   divergence but not faithful measurements. The fix is upstream
   in `dl-sim`: `CostModel` and `EvalParams` need a tip field, and
   `CycleRecord` / `LedgerEntry` need to carry per-cycle tip and
   per-cycle PnL distributions.
3. Median/P95 percentile tracking in `LedgerSummary` (06-02 honest
   deferral). Required to make `MedianWinnerPnlSol` and
   `P95WinnerPnlSol` engine aggregates return real values.

**ROADMAP scope this plan delivers:**
- Metrics dashboards: opps/sec, detection latency, hit rate, drift,
  paper PnL.
- Config-driven strategy params: no recompile to retune.

The plan intentionally does NOT deliver: multi-pool/multi-DEX
scale-up, v1.0 release docs. Those are plan 07-02.

## Why this split

Multi-pool/multi-DEX scale-up (07-02) requires touching the
detection graph (per-DEX node labeling, multi-DEX route enumeration)
and the v1.0 release doc. Both are larger changes that benefit
from a clean cut after 07-01 lands and the engine has been run for
real over a long session with metrics flowing.

07-01 is "make the engine operable and observable." 07-02 is
"ship v1.0 with scale-up + docs." Different size, different
review, different risk profile.

## Goal

Make the engine operable over multi-day sessions, with:
1. **Config-driven params** — every `EvalParams` field, every cost
   model component, every Jito bundle config, every capture
   setting, every recon option overridable via TOML or env vars
   without recompile.
2. **Metrics emission** — a `MetricsSink` trait that any number of
   backends can implement. Default backend: structured `tracing`
   events with stable field names (so Prometheus/OTel adapters
   can scrape them in 07-02). The plan ships the trait, the
   `tracing` adapter, and an in-memory test sink.
3. **Ledger write from dry-run and live capture** — `dl-app`
   `run_dry_run` and `run_capture` both produce a `DLD-LDG1`
   ledger when `DL_LEDGER_PATH` is set. Closes the 05-02 AC-6
   deferral.
4. **Per-cycle tip in the ledger** — extends `LedgerEntry` (or a
   sidecar) to carry per-cycle tip and per-cycle PnL distribution.
   Unblocks the three placeholder `engine_aggregate()` mappings
   in `dl-recon::onchain`.
5. **Median/P95 in `LedgerSummary`** — `LedgerSummary::from_entries`
   extends to also return `median_conservative_e_pnl` and
   `p95_conservative_e_pnl` as `i128`. Unblocks the same
   placeholders.

## Scope

In scope:
- `crates/dl-core/src/config.rs` (new): `EngineConfig` struct,
  `EngineConfig::load_toml(path)` + `EngineConfig::from_env()`.
  Pure data + validation. No business logic.
- `crates/dl-app/src/config_loader.rs` (new): wires `EngineConfig`
  to `EvalParams`, `CostModel`, capture settings, recon settings.
  Single source of truth.
- `crates/dl-core/src/metrics.rs` (new): `MetricsSink` trait,
  `Counter` / `Gauge` / `Histogram` types, `MetricsRegistry` for
  fan-out. Integer-only values (no `f64` in the counters; rate is
  a separate computed field).
- `crates/dl-app/src/metrics_tracing.rs` (new): `MetricsSink` impl
  that emits `tracing::info!` events with stable field names.
- `crates/dl-ledger/src/entry.rs` (modify): add `tip_lamports: u64`
  field. Bump `LEDGER_SCHEMA_VERSION` from 2 to 3. Old v2 readers
  must error gracefully (per the schema policy: "opening a v3
  file with a v2 reader is a hard error").
- `crates/dl-ledger/src/summary.rs` (modify): add `median` and
  `p95` computation. Integer-only (no `f64`).
- `crates/dl-app/src/main.rs` (modify): `run_dry_run` and
  `run_capture` accept `DL_LEDGER_PATH`; if set, open a
  `LedgerWriter` and route every `CycleRecord` through it.
- `crates/dl-recon/src/onchain.rs` (modify):
  `engine_aggregate()` for `MeanTipLamports`, `TipAsPctOfMev`,
  `Median/P95WinnerPnlSol` now return real values from
  `ReconReport::tip_lamports` and `LedgerSummary::median/p95`.
- `crates/dl-recon/src/pipeline.rs` (modify): `ReconReport` gains
  `total_tip_lamports: u64` and `p50/p95_conservative_e_pnl: i128`.
- Metrics integration: every site that produces a number a
  dashboard would want emits a `metrics.counter!` / `metrics.gauge!`
  call. Sites: capture (events/sec, write_failures), detection
  (cycles_found, per-leg), simulation (cycles_evaluated,
  would_trade, would_not_trade), ledger (entries_written,
  total_e_pnl).

Out of scope (plan 07-02):
- Multi-DEX routing (Orca, Meteora decoders).
- Prometheus / OTel adapter (the `MetricsSink` trait is enough;
  the adapter is a thin wrapper).
- v1.0 release documentation.
- Live Jito API client (still an open gate from 06-02).

## Research gate

**None.** All claims in this plan are about extending existing
in-workspace code. No external API contracts or third-party
protocols are involved. The metrics trait is an internal pattern;
the ledger schema bump is internal.

## Acceptance criteria

- **AC-1**: `EngineConfig` loads from TOML file. Field count
  matches `EvalParams + CostModel + capture + recon` (≥20
  fields). Missing fields fall back to `EvalParams::conservative_default()`.
  Invalid values (e.g., `tip_lamports < 0` on a `u64` field)
  return a typed `ConfigError` rather than panicking. Verified
  by `tests/config_loads_and_validates.rs`.

- **AC-2**: Every `EvalParams` and `CostModel` field is
  overridable via `EngineConfig`. Verified by a roundtrip test
  that constructs a non-default `EngineConfig`, applies it,
  asserts the resulting `EvalParams` differs from
  `conservative_default()`. ≥ 5 fields must change.

- **AC-3**: `MetricsSink` trait is dyn-compatible (no generics in
  the trait method signatures). `Counter` / `Gauge` / `Histogram`
  are integer-only. Verified by
  `tests/metrics_sink_trait_is_object_safe.rs` and
  `tests/metrics_no_floats.rs`.

- **AC-4**: The `tracing` adapter implements `MetricsSink` and
  emits one `tracing::info!` per metric update with stable
  field names (`opps_per_sec`, `detection_latency_ms`, etc.).
  Verified by capturing `tracing` output and asserting the
  field names exist.

- **AC-5**: `dl-app` `run_dry_run` and `run_capture` both honor
  `DL_LEDGER_PATH`. If set, every detected cycle is written
  to the ledger. If unset, behavior is unchanged. Verified by
  an integration test that runs `run_dry_run` with
  `DL_LEDGER_PATH` set to a temp file and asserts the file
  starts with `DLD-LDG1` and contains ≥1 entry.

- **AC-6**: `LedgerEntry` schema bumped to v3. v2 readers reject
  v3 files with `LedgerError::SchemaMismatch { found: 3, expected: 2 }`.
  v3 readers reject v2 files with `found: 2, expected: 3`.
  Round-trip write/read of v3 produces identical bytes.
  Verified by `tests/ledger_schema_v3_round_trip.rs`.

- **AC-7**: `LedgerSummary` gains `median_conservative_e_pnl` and
  `p95_conservative_e_pnl` (both `i128`). Computed in
  integer-only fashion (no sort + lookup, no `f64`).
  Verified by a test with a known input distribution and
  known median/p95.

- **AC-8**: `dl-recon::onchain::engine_aggregate()` for
  `MeanTipLamports`, `TipAsPctOfMev`, `MedianWinnerPnlSol`,
  `P95WinnerPnlSol` no longer returns placeholders. Returns
  real values computed from `ReconReport::total_tip_lamports`,
  `LedgerSummary::median`, `LedgerSummary::p95`. Verified by
  the existing `tests/onchain.rs` empty-report-divergence test
  plus a new test where a populated report produces the
  expected aggregates.

- **AC-9**: Float-free CI guard still passes after the schema
  bump and the per-cycle tip addition. `tests/floats.rs` covers
  `dl-recon` and `dl-recon-overfit`; new guards cover
  `dl-core::config`, `dl-core::metrics`,
  `dl-app::metrics_tracing`. Total float-free guards in
  workspace: ≥ 8 (was 6).

- **AC-10**: `cargo build --workspace`, `cargo test --workspace`,
  `cargo fmt --all`, `cargo clippy --workspace --all-targets -- -D warnings`
  all clean. Test count ≥ 320 (was 280; +40 expected from the
  new AC-1, AC-3, AC-4, AC-6, AC-7, AC-8, AC-9 tests plus
  expanded coverage on existing modules).

## Risks

- **Risk 1: Schema bump breaks 05-02 ledger fixtures.** The
  existing ledger test fixtures (Phase 5 roundtrip) write v2
  files. Bumping to v3 means those fixtures need to be
  regenerated. Mitigated by a one-shot migration script
  embedded in the test that converts v2 → v3 (or just deletes
  the v2 fixtures and lets the v3 fixtures replace them).

- **Risk 2: `Median/P95` integer-only computation is tricky.**
  Sorting 1M `i128` values is O(n log n) but allocates a
  Vec. For a paper ledger, this is fine (≤100K entries in
  any reasonable session). Mitigated by using
  `select_nth_unstable` for the median and a single pass for
  p95 via a quickselect.

- **Risk 3: Config sprawl.** 20+ fields × 4 modules (eval,
  cost, capture, recon) is a lot of configuration surface.
  Mitigated by grouping under named sub-configs
  (`[engine.eval]`, `[engine.cost]`, `[engine.capture]`,
  `[engine.recon]`) and providing a `dl-app config print`
  subcommand that dumps the active config for debugging.

- **Risk 4: Metrics `tracing` adapter emits one event per
  update.** For high-frequency events (opps/sec on a busy
  slot), this could flood the log. Mitigated by allowing
  per-metric sample rates (default 1Hz, opt-in for higher).

- Prometheus / OTel adapter (the `MetricsSink` trait is the
  integration point; the adapter itself is a thin wrapper).
- v1.0 release documentation.
- Live Jito API client.
- DSR formula PDF cross-check (06-02 deferral).
- Closed-loop `calibrate()` that re-runs the engine.
- Real `ReconReport::divergences` population against a real
  source-ledger.
