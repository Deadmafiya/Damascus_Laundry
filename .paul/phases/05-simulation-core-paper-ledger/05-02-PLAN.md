<objective>
## Goal
Build the **paper ledger** that records every evaluated opportunity in v1.0
paper-trading. The ledger is an append-only length-prefixed bincode file
(8-byte magic `DLD-LDG1` + u32 schema version + frames of `LedgerEntry`),
mirroring the Phase 2 capture format. Each entry contains the cycle identity
(deterministic hash of the leg sequence), the Phase-4 `NetProfit`, the
Phase-5 dual-bound `EvalOutcome` (optimistic + conservative), the trading
decision (`WouldTrade` iff `conservative.e_pnl > 0`), and a deterministic
monotonic sequence number. A `LedgerReader` exists for replay; a
`LedgerSummary` produces aggregate counts. **No PnL math, no calibration,
no portfolio state** — that is Phase 6.

## Purpose
v1.0 is paper-trading. Without a ledger, paper PnL is a *current run*
number, lost on restart. The ledger gives us:
- **Replay-ability** — a recorded session can be re-evaluated under different
  conservative settings (Phase 6 calibration will re-score the same entries
  with new p_win / p_land defaults).
- **Auditability** — every "would-trade" decision is on disk, attributable
  to the exact opportunity that produced it.
- **Determinism** — given a captured feed and a fixed set of `EvalParams`,
  the ledger's contents are byte-identical across machines (AC-1).

## Output
- `dl-ledger` crate filled out: `LedgerEntry`, `Decision`, `LedgerWriter`,
  `LedgerReader`, `LedgerSummary`, `LedgerError`, format spec, schema
  constants, `LedgerHash` (deterministic cycle id).
- A wiring test in `dl-app` that runs the dry-run replay path *and* writes
  the resulting entries to a ledger file (still no live submission, still
  no detection wiring — this exercises the write/read/aggregate path
  only).
- Unit + property tests; int-only invariant preserved (no `f32`/`f64`).
- All gates green; no new external dependencies.
</objective>

<acceptance_criteria>

## AC-1: Append-only file format is a strict mirror of the Phase 2 capture format

Given a fresh ledger file
When the `LedgerWriter::new(path)` constructor runs
Then the file contains:
  - 8 bytes magic `b"DLD-LDG1"`
  - 4 bytes u32 LE schema version (`LEDGER_SCHEMA_VERSION = 2`)
…and the layout is documented as ASCII art in the module's doc-comment
exactly as `dl_feed::capture` does for the capture format, with a
`format_spec()` function that returns the spec string and a round-trip
test asserting the spec text mentions every layout field. The round-trip
test exercises `write N entries → reopen with reader → read N entries →
assert byte-equality`.

## AC-2: `LedgerEntry` is bincode-serializable and contains the dual-bound EV

Given an evaluated opportunity (a `Cycle` and the `EvalOutcome` from
`dl_sim::ev::evaluate`)
When `LedgerEntry::from_evaluated(cycle, net_profit, outcome, seq)` runs
Then the returned struct contains:
  - `seq: u64` (monotonic, caller-supplied)
  - `cycle_hash: LedgerHash` (deterministic u64 over leg pubkeys + directions)
  - `net: NetProfit` (bincode-encodable; NetProfit is already an integer-only struct)
  - `optimistic: ExpectedValue`
  - `conservative: ExpectedValue`
  - `decision: Decision` (derived: `WouldTrade` iff `conservative.e_pnl > 0`, else `WouldNotTrade`)
  - `entry_id: u64` (= seq, also serves as canonical "this opportunity at this seq" id)
…and the round-trip test confirms bincode encode → decode returns a
struct equal to the original. No floats anywhere.

## AC-3: `LedgerHash` is a deterministic u64 over the cycle leg sequence

Given a `Cycle` with N legs (mint in/out, pool pubkey, direction)
When `LedgerHash::from_cycle(&cycle)` runs
Then it returns a `u64` derived by feeding each leg's `(pool_pubkey,
direction)` pair into a deterministic 64-bit hash (FNV-1a 64 or
splitmix64-style mixer — implementation choice as long as it is
deterministic, fixed-width, and `Copy`). Same cycle → same hash.
Different cycle (any leg differs) → almost certainly different hash
(verified by proptest over random cycles).

## AC-4: `LedgerReader` is a frame-by-frame iterator that errors on corruption

Given a ledger file written by `LedgerWriter`
When `LedgerReader::open(path)?` runs
Then it validates the magic (`DLD-LDG1`) and schema version (== 2) at open
time, returning `LedgerError::BadMagic` or `LedgerError::SchemaMismatch`
on mismatch. Subsequent `read_entry() -> Result<Option<LedgerEntry>>`
calls return the next entry until EOF, at which point they return `Ok(None)`.
A truncated frame (declared `payload_len` larger than the file) returns
`LedgerError::Truncated`. Bincode decode failures return `LedgerError::Bincode(_)`.

## AC-5: `LedgerSummary` aggregates decisions and EVs without floats

Given a sequence of `LedgerEntry`s (from a reader or an in-memory Vec)
When `LedgerSummary::from_entries(&[LedgerEntry])` runs
Then it returns:
  - `total: u64`
  - `would_trade: u64`
  - `would_not_trade: u64`
  - `sum_optimistic_e_pnl: i128` (sum across all entries, computed via `i128::checked_add`)
  - `sum_conservative_e_pnl: i128`
  - `sum_conservative_p_land: u128` (ppm, sum as integer ppm)
…and the in-memory summary over a sequence equals the per-entry
summary over the same sequence read from a file (round-trip property).

## AC-6: dl-app dry-run path writes a ledger when `DL_LEDGER_PATH` is set

Given `DL_DRY_RUN=1` and `DL_LEDGER_PATH=/tmp/x.ledger`
When `cargo run -p dl-app` runs
Then:
  - The existing dry-run replay runs to completion (no regressions).
  - A ledger file is created at `/tmp/x.ledger` with the correct magic +
    schema header.
  - The ledger contains 0 entries (the dry-run path has no detection
    wiring in 05-02 — that's Phase 6). The point of 05-02 is to prove
    the *write path* opens and the *read path* reads an empty ledger
    back without error.
  - Without `DL_LEDGER_PATH`, the dry-run path is unchanged (no
    ledger file is touched).
…and the integration test asserts both behaviors.

## AC-7: Int-only invariant preserved in dl-ledger

Given the float-free CI guard pattern from Phase 2-5
When `cargo test -p dl-ledger --test int_only_no_fractional` runs
Then it scans every `.rs` file under `crates/dl-ledger/src/` for
`f32`, `f64`, and the substring "float" / "Float" (case-insensitive)
and panics with the file + line if any are found. This guard must be
green in the 05-02 delivery.

- **In scope:**
  - `dl-ledger` crate: `LedgerEntry`, `Decision`, `LedgerHash`, `LedgerWriter`,
    `LedgerReader`, `LedgerSummary`, `LedgerError`, format spec, schema constants.
  - `dl-app` dry-run path: optional `DL_LEDGER_PATH` opens a writer (writes
    0 entries in 05-02 — detection wiring is Phase 6).
  - Int-only CI guard for `dl-ledger`.
  - One in-binary integration test exercising the dry-run + ledger path.
- **Out of scope (deferred):**
  - **No PnL math, no portfolio state, no realized/unrealized tracking** —
    Phase 6.
  - **No detection-driven write loop** — Phase 6 (needs the detection +
    sim pipeline to be wired in `dl-app`).
  - **No calibration / re-score** — Phase 6 reads the ledger with new
    `EvalParams` and recomputes the conservative EV.
  - **No `Display` impl on `LedgerSummary`** — Phase 6 builds the
    human-readable report.
  - **No `From<DetectError>` for `LedgerError`** — not needed until
    Phase 6 wires detection failures into ledger writes.
  - **No async writer** — `LedgerWriter` is sync over `std::io::Write` to
    match the `dl-feed` capture pattern. (Phase 7 may add a buffered
    variant if the writer becomes a bottleneck; not needed in v1.0.)
