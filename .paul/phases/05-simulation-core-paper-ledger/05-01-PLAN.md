<objective>
## Goal
Build the **pessimistic-by-default simulation core** (`dl-sim::ev`): take a Phase-4
`NetProfit` for a detected cycle and produce an **expected-value** estimate via the
multiplicative decomposition
`E[PnL] = p_detect × p_win × p_land × (gross − costs) − E[failed_costs]`,
where every probability is a fixed-point (parts-per-million) multiplier, none is 1.0, and
`p_win` *decreases* as the opportunity gets richer (winner's curse). Report an
**optimistic and a conservative bound** for every opportunity; downstream acts only on the
conservative one.

## Purpose
This is the accuracy heart of v1.0 (Principles 5, 6, 9 of the simulation research). Phase 4
gives the per-cycle *gross-minus-costs* math assuming we are alone and always land. Phase 5
applies the three pessimistic multipliers that turn that fantasy number into an honest
expected value. Without this, the bot's paper PnL is optimistic by construction and lies.

## Output
A new `ev` module in `dl-sim` with: a fixed-point `Prob` type (ppm), a `LatencyBudget`
→ `p_land` model, a richness-decreasing `p_win` model, a `FailedCostModel`, the
`ExpectedValue { optimistic, conservative }` result, and the top-level
`evaluate(net_profit, params) -> EvalOutcome`. Pure functions, deterministic, integer-only,
unit + property tested. **No ledger, no I/O** (that is 05-02).
</objective>

<acceptance_criteria>
## AC-1: Fixed-point probability type, no floats
Given the engine's integer-only invariant
When probabilities are represented and multiplied
Then a `Prob` type stores probability as parts-per-million (`u32`, 0..=1_000_000), all
combination uses `dl-core::fixed::mul_div_floor` (or equivalent checked integer math), and
the `dl-sim` float-free CI guard (`tests/fixed_point_no_fractional.rs`) still passes with
the new `ev` module included

## AC-2: Multiplicative EV decomposition is correct and pessimistic
Given a Phase-4 `NetProfit` with a positive `net_profit` (gross − costs > 0)
When `evaluate` runs with non-degenerate probabilities (each < 1_000_000 ppm)
Then the conservative `E[PnL]` is strictly less than the raw `net_profit`, equals
`p_detect ⊗ p_win ⊗ p_land ⊗ (gross − costs) − E[failed_costs]` computed in `i128`, and a
`NetProfit` whose `profitable == false` (or `OptimalInput::NoTrade`) yields a non-positive
EV without panicking

## AC-3: p_win decreases with opportunity richness (winner's curse)
Given two cycles identical except that cycle B has a larger `net_profit_bps` than cycle A
When `p_win` is computed for each at the same competition setting
Then `p_win(B) <= p_win(A)` (strictly < once B crosses the richness threshold), encoding
that fatter spreads attract more/faster competitors — verified by a property test over
random richness pairs

## AC-4: Latency maps to p_land monotonically, anchored to Jito timing
Given a `LatencyBudget` (t_detect + t_decide + t_build + t_network + t_auction in ms)
When `p_land` is derived
Then larger total latency yields smaller (or equal) `p_land`, the Jito auction component is
anchored to the ~200 ms window / 50 ms ticks (documented constants), and `p_land` is clamped
to `[0, 1_000_000]` ppm — verified by a monotonicity property test

## AC-5: Dual bounds reported together, conservative ≤ optimistic
Given any evaluated opportunity
When `evaluate` returns an `EvalOutcome`
Then it carries both an `optimistic` and a `conservative` `ExpectedValue`, with
`conservative.e_pnl <= optimistic.e_pnl` always, and the optimistic bound corresponds to
the documented "you detect instantly, win, land at min tip" assumption set while the
conservative bound uses the full haircut stack — both surfaced, never just one
</acceptance_criteria>

  integer-only — no `f32`/`f64`, no `as f64`. Add unit tests: `from_ppm` bounds, `combine`
  identity/zero/associativity-ish, `apply_to` on positive and negative values.

<task type="auto">
  <name>Implement p_win (winner's-curse) and p_land (latency) models</name>
  <files>crates/dl-sim/src/ev.rs</files>
  <action>
  In `ev.rs` add two deterministic, integer-only models. Default constants live in named
  `const`s with doc comments citing the research (calibration of exact curves is deferred to
  Phase 6 — these are *structurally* correct, conservatively-tuned placeholders).
  - `struct CompetitionParams { base_win_ppm: u32, richness_threshold_bps: i32, decay_ppm_per_bps: u32 }`
    and `fn p_win(net_profit_bps: i32, params: &CompetitionParams) -> Prob`:
    starts at `base_win_ppm` and **decreases** as `net_profit_bps` exceeds
    `richness_threshold_bps` (linear decay by `decay_ppm_per_bps`, saturating at 0). Below
    the threshold, return `base_win_ppm`. This inverts naive optimism (Principle 6).
  - `struct LatencyBudget { t_detect_ms, t_decide_ms, t_build_ms, t_network_ms, t_auction_ms: u32 }`
    with `total_ms()`, plus a `JITO_AUCTION_MS = 200` and `JITO_TICK_MS = 50` documented
    constant. `fn p_land(budget: &LatencyBudget, params: &LandingParams) -> Prob`:
    `p_land` is `1e6` at/under a `grace_ms`, then **decreases monotonically** with total
    latency (e.g. linear or step-per-tick decay), clamped to `[0, 1e6]`. Document that the
    real "advance state to projected landing slot and re-score" model is Phase 6 (needs
    pool-bearing captures); this haircut is the v1.0 stand-in.
  Unit tests: p_win at/above/below threshold; p_win monotonic non-increasing in richness;
  p_land monotonic non-increasing in latency; both clamp correctly.
  </action>
  <verify>`cargo test -p dl-sim ev::` passes including the p_win/p_land unit tests</verify>
  <done>AC-3 + AC-4 satisfied: richness-decreasing p_win, latency-decreasing p_land anchored to Jito timing</done>
</task>

<task type="auto">
  <name>Implement failed-cost model + EV decomposition + dual bounds</name>
  <files>crates/dl-sim/src/ev.rs</files>
  <action>
  Add the cost-of-losing and the top-level evaluator (Principles 5, 7, 9).
  - `struct FailedCostModel { attempts_per_win: u32, per_attempt_lamports: u64, path: SubmitPath }`
    where `enum SubmitPath { Spam, JitoBundle }`. `fn expected_failed_cost(&self) -> u128`:
    - `Spam`: every *included* tx (incl. reverts) pays the base sig fee → roughly
      `attempts_per_win × per_attempt_lamports`.
    - `JitoBundle`: failed bundles do NOT land → no tip on losses; failed-cost is ~0 (only
      the won bundle's tip, already in Phase-4 costs). Document this asymmetry (Principle 7).
  - `struct ExpectedValue { e_pnl: i128, p_detect: Prob, p_win: Prob, p_land: Prob, expected_failed_cost: u128 }`.
  - `struct EvalParams { p_detect: Prob, competition: CompetitionParams, latency: LatencyBudget, landing: LandingParams, failed: FailedCostModel }`
    plus a `EvalParams::optimistic()` (p_detect=1e6, no richness decay, latency≤grace so
    p_land=1e6, failed cost=0) and `EvalParams::conservative_default()` (full haircuts).
  - `fn evaluate(net: &NetProfit, optimistic: &EvalParams, conservative: &EvalParams) -> EvalOutcome`
    returning `struct EvalOutcome { optimistic: ExpectedValue, conservative: ExpectedValue }`.
    Core math, in `i128`:
    `gross_minus_costs = net.net_profit` (already signed net of all Phase-4 costs);
    `haircut = p_detect.combine(p_win).combine(p_land)`;
    `e_pnl = haircut.apply_to(gross_minus_costs) - (expected_failed_cost as i128)`.
    If `net.profitable == false`, EV must be ≤ 0 (do not haircut a loss upward; apply
    probabilities to the loss magnitude too, or simply pass the negative through — choose so
    that a losing cycle never reports positive EV; document the choice).
  Unit tests: optimistic ≥ conservative; conservative < raw net for a profitable cycle;
  losing cycle → EV ≤ 0; spam vs jito failed-cost asymmetry.
  </action>
  <verify>`cargo test -p dl-sim ev::` passes; assert in a test that for a sample profitable NetProfit, `conservative.e_pnl < net.net_profit` and `conservative.e_pnl <= optimistic.e_pnl`</verify>
  <done>AC-2 + AC-5 satisfied: multiplicative EV, failed-cost asymmetry, dual bounds ordered</done>
</task>
