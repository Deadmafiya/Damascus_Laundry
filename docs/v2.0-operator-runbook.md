## Phase 1c — mainnet-paper (0.001 SOL/day cap)

Pre-flight:
- [ ] **Build BPF binary** — from repo root:
  ```bash
  cd crates/dl-assert-program
  cargo build-sbf --release
  # → target/deploy/dl_assert_program.so
  ```
  Requires Solana SDK install (`sh -c "$(curl -sSfL https://release.anza.xyz/v2.0/install)"`).

- [ ] **Deploy to devnet first** (record the program ID):
  ```bash
  solana program deploy target/deploy/dl_assert_program.so \
      --url devnet \
      --keypair ~/.config/solana/id.json \
      --program-id dl_assert_dev.json   # optional, for deterministic ID
  ```
  → Record the deployed program ID (e.g. `DLasrt…xyz`). This is
  `LiveConfig.assert_program_id` in dl-app.

- [ ] **Run devnet e2e test** (Tier 1 in `docs/live-runbook.md`):
  - Generate devnet keyfile:
    `DL_LIVE_MODE=devnet dl-signer generate --out ~/.damascus/devnet-keyfile.json`
  - Fund devnet wallet: `solana airdrop 1 <DEVNET_PUBKEY>`
  - Fund vault PDA once per session (per atomicity ADR):
    ```
    VAULT=$(dl-assert-sdk derive-vault-pda <SIGNER> <ASSERT_PROGRAM>)
    solana transfer --from <DEVNET_KEYPAIR> $VAULT <SIGNER_LAMPORTS>
    ```
  - Run with `DL_LIVE_MODE=mainnet-paper ./dl-app run --mode devnet \
        --assert-program-id <DEVNET_PROGRAM_ID> \
        --daily-cap-sol 100 --per-bundle-cap-sol 1`
  - Confirm ≥1 cycle lands with `dl-app recon` showing predicted vs
    realized net-PnL within ±0.001 SOL. The canonical one-liner
    (resolves a `bundle_id` to its capture and runs the recon) is:
    ```bash
    scripts/recon_bundle.sh <bundle_id>      # operator one-liner
    ```
    Output: JSON to stdout; schema in
    `docs/recon/recon-json-schema.md`. The script exits non-zero
    if the gap exceeds the ±0.001 SOL tolerance (calibration event,
    feeds into DAM-35).

- [ ] **Redeploy to mainnet-beta** (only after devnet works):
  ```bash
  solana program deploy target/deploy/dl_assert_program.so \
      --url mainnet-beta \
      --keypair <MAINNET_DEPLOYER>
  ```
  → Record the mainnet program ID. This is your `assert_program_id`
    for all mainnet runs.

- [ ] **Verify the deploy** with the DAM-59 verification script
  (asserts program id matches config, account is Executable, data
  account is intact, on-chain bytecode sha256 matches local .so):
  ```bash
  scripts/verify_assert_program_deploy.sh \
      --cluster mainnet-beta \
      --assert-program-id <MAINNET_PROGRAM_ID> \
      --so target/deploy/dl_assert_program.so \
      --keypair <MAINNET_DEPLOYER>
  ```
  Exits 0 only if the program is correctly deployed. **Do not start
  `dl-app` with this `assert_program_id` until the script is green.**
  Re-run with `--skip-deploy` after any manual upgrade to re-check
  without re-uploading.

- [ ] **Generate mainnet hot-wallet keyfile** (separate from devnet):
  `DL_LIVE_MODE=devnet dl-signer generate --out ~/.damascus/mainnet-keyfile.json`

- [ ] **Fund the hot wallet** (manual `solana transfer` from cold
  wallet — locked decision #3, no auto-rebalance):
  ```
  # mainnet-paper cap = 0.001 SOL/day → fund 0.005 SOL (5 days runway)
  solana transfer --from <COLD_KEYPAIR_OR_LEDGER> \
      <MAINNET_HOT_PUBKEY> --amount 0.005
  ```

- [ ] **Verify the cap holds** at runtime:
  ```bash
  DL_LIVE_MODE=mainnet-paper \
  DL_SIGNER_PASSPHRASE='...' \
  cargo run --release -p dl-app -- run --mode mainnet-paper \
      --assert-program-id <MAINNET_PROGRAM_ID> \
      --keyfile ~/.damascus/mainnet-keyfile.json \
      --daily-cap-sol 0.001 --per-bundle-cap-sol 0.00001 \
      --landing-timeout-ms 30000
  ```
  The `0.001 SOL/day` is hard-coded in `dl-signer` (`mainnet-paper`
  case in `cap.rs`); `--daily-cap-sol 0.001` matches it. The
  per-bundle cap is the Jito tip ceiling (`--per-bundle-cap-sol`
  default ≈ 10_000 lamports = 0.00001 SOL).

- [ ] **Cap never exceeds** — verify with the recon CLI:
  ```bash
  # Replay the capture that produced today's bundles and ask the
  # harness to compare against the anchor dataset.
  cargo run --release -p dl-app -- recon \
      --capture ./captures/$(date +%Y%m%d).bincode \
      --anchors  ./anchors/$(date +%Y%m%d).jsonl \
      --report-json ./recon/$(date +%Y%m%d).json
  ```
  Expected: `summary.would_trade * per-bundle-tip ≤ 0.001 SOL` and
  the report JSON parses against `docs/recon/recon-json-schema.md`.
  Anything more than the daily cap = bug, halt. (The recon CLI is
  the working `dl-recon` surface today; a thin `dl-recon` shim that
  delegates to `dl-app recon` is tracked in DAM-60 follow-ups.)

- [ ] **Gate to Phase 2**: ≥1 bundle lands on mainnet + reconciles
  within ±0.001 SOL (gate defined in `docs/recon/recon-json-schema.md`)
  + no cap/kill-switch violations + no key-handling defect. Document
  the bundle_id + slot + realized PnL, then file DAM-60 with the
  recon JSON attached and tag it for the CTO's ±0.001 SOL sign-off.

## Phase 1d — tiny mainnet (<0.5 SOL/day cap)

Pre-flight:
- [ ] Phase 1c gate cleared (above).
- [ ] Daily cap raised:
  ```
  --daily-cap-sol 0.5 --per-bundle-cap-sol 0.05
  ```
- [ ] Hot wallet topped up: `solana transfer <COLD> <HOT> --amount 2.5`
  (5× daily cap so a single forgot-to-top-up day doesn't kill the run).

Run loop:
- [ ] `DL_LIVE_MODE=mainnet ./dl-app run --mode mainnet \
       --daily-cap-sol 0.5 --per-bundle-cap-sol 0.05 …`
- [ ] Run ≥24 h unattended. `tail -f trader.log` + Prometheus on
  `:9090/metrics`.
- [ ] Recon after the run:
  ```bash
  cargo run --release -p dl-recon-overfit -- report \
      --in ./recon-$(date +%Y%m%d).json
  ```
  Expected: Deflated Sharpe ≥ 0 on the landed sample (sanity, not
  statistical proof).

- [ ] **Replace v1.2.0 latency probe** (the placeholder-sig stub):
  - The replacement is at `crates/dl-executor/src/landing.rs`
    (`HttpJitoClient::poll_landing`) + a small `crates/dl-app/src/bin/dl-latency-probe.rs`
    binary that constructs a real signed no-op tx via the executor
    and times it through Jito. Wire it to replace `latency_probe.py`'s
    HTTP path. (Currently the placeholder-sig probe is what
    `latency_probe.py` calls — to be deprecated once the real
    one is in place.)
  - For now: stop calling `latency_probe.py`. Run the new probe
    manually:
    ```
    cargo run --release -p dl-app --bin dl-latency-probe \
        --signing-keyfile ~/.damascus/mainnet-keyfile.json \
        --probe-count 20
    ```
    Logs submission-to-landing latencies; rolling median replaces
    `snap.rpc.latency_ms` in `damascus_laundry_dashboard`.
  - Pre-flight before this section: confirm `latency_probe.py`
    is not being called from anywhere in the repo.
    ```
    grep -rn latency_probe crates/ dashboard/ scripts/ 2>/dev/null
    ```
    Expected: zero matches. Anything else = the old probe is
    still in the path, investigate before continuing.

Gate to Phase 2: ≥3 profitable bundles across ≥24 h + positive
net PnL + reconciles.

## Phase 2 — calibration (the meaty coding work — see agent)

This phase is the *next* thing the agent will code. Operator
expertise here is mostly review. Key infra changes you'll see:

- Paid RPC upgrade (recommended ~$50–250/mo Helius/Triton/QuickNode

## Phase 3 — 24/7 reliability (mostly code, one systemd unit)

Operator step (one-time):
- [ ] **Write systemd unit** at `/etc/systemd/system/dl-app.service`:
  ```ini
  [Unit]
  Description=damascus_laundry live trader
  After=network-online.target

  [Service]
  Type=simple
  User=<your-user>
  EnvironmentFile=/home/<your-user>/.damascus/dl-app.env
  ExecStart=/home/<your-user>/.cargo/bin/dl-app run \
      --mode mainnet-paper --keyfile /home/<your-user>/.damascus/mainnet-keyfile.json \
      --assert-program-id <MAINNET_PROGRAM_ID>
  Restart=on-failure
  RestartSec=10
  StandardOutput=journal
  StandardError=journal

  [Install]
  WantedBy=multi-user.target
  ```
  Then `systemctl daemon-reload && systemctl enable --now dl-app`.

- [ ] Prometheus alert rules (sample) at `prometheus/rules/dl-app.yml`:
  - `dl_jito_submit_failures_total > 3 in 5m`
  - `dl_simulate_rejected_total / dl_jito_submit_total > 0.5`
  - `dl_stop_file_present == 1`
  - `dl_killswitch_consecutive_losses >= 3`

Gate to Phase 4: survives a multi-day unattended run, including
induced `kill -9` on RPC mid-trade.
