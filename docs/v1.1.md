## 0. What is damascus_laundry?

`damascus_laundry` is a Solana MEV (Maximal Extractable Value) paper-trading
engine written in Rust. As of v1.0.0 (tagged and shipped on 2026-06-19), it:

1. **Watches** real-time Solana DEX (decentralized exchange) pool accounts
   through a WebSocket RPC connection (or replay from a recorded file).
2. **Decodes** the raw on-chain account bytes for three AMM families:
   Raydium AMM v4, Orca Whirlpool, and Meteora DLMM.
3. **Detects** atomic DEX-DEX arbitrage cycles using Bellman-Ford
   negative-cycle search on a token-priced graph.
4. **Simulates** each cycle with a conservative-by-default EV
   (expected value) decomposition: `E[PnL] = p_detect Г— p_win Г— p_land Г—
   (gross в€’ costs) в€’ E[failed_costs]`, where each probability is
   integer-scaled, each cost is lamport-exact, and every calculation
   uses `u128` fixed-point arithmetic (no floating-point in the value path).
5. **Records** every evaluated cycle in a v3 paper ledger
   (`DLD-LDG1` magic) вЂ” never submits a real transaction.

**What it does NOT do at v1.0.0:** submit transactions, sign anything,
hold a wallet, connect to a wallet, pay gas, pay tips, race for
mempool priority, or earn real SOL. It is a *signal generator* that
proves the math works, not a *trader*.

The user goal for v1.1 is to close that gap: from paper simulator to
millisecond-scale live trader on Solana mainnet.

## 2. The `graphify-out/` folder

The `graphify-out/` folder is the **code map + audit** layer. It's a
machine-generated knowledge graph that lets a new operator navigate
the codebase, trace dependencies, and verify that the code does what
the planning docs claim. It is rebuilt from source by a tool called
`graphify`; it's not hand-written.

The folder is **not** the source of truth for *why* a design decision
was made (that's `.paul/`, Section 1). It's the source of truth for
*what the code actually does* вЂ” including drift from the planning docs.

### File inventory

| Path | Role |
|------|------|
| `graphify-out/graph.json` | Raw graph data (NetworkX node-link format). ~1.4 MB at v1.0.0. Every Rust symbol as a node, every call/import as an edge. |
| `graphify-out/graph.html` | Interactive browser-viewable graph. Color-coded by community, drag-to-zoom, click-for-neighbors. ~1.2 MB. |
| `graphify-out/GRAPH_REPORT.md` | Human-readable audit report. ~21 KB / 408 lines at v1.0.0. Contains: corpus check, summary stats, god nodes, surprising connections, suggested questions, per-community listings. |
| `graphify-out/manifest.json` | Per-file mtime + hash. Used by `graphify update` to know what changed. ~29 KB. |
| `graphify-out/cost.json` | Cumulative LLM token cost across runs. v1.0.0: 0 tokens (semantic subagent API was unavailable; the host agent did the extraction inline). |
| `graphify-out/cache/ast/` | Per-file AST extraction cache. Reused on `--update` runs. |
| `graphify-out/cache/semantic/` | Per-file semantic extraction cache (same purpose). |
| `graphify-out/.graphify_python` | Internal: path to the Python interpreter graphify was installed with. Don't edit. |
| `graphify-out/.graphify_root` | Internal: the scan root (the project directory). Don't edit. |
| `graphify-out/.graphify_labels.json` | Internal: human-chosen labels for the communities. 80+ entries. |

### The graph structure

The graph is a NetworkX undirected graph with metadata on every node
and edge.

- **Node** = a Rust symbol or a `file_type` enum (code, document,
  paper, image, rationale, concept). Each node has:
  - `id`: deterministic snake_case from the file path + symbol name
    (e.g. `crates_dl_sim_src_ev_rs_p_land`)
  - `label`: human-readable name (e.g. `p_land()`)
  - `file_type`: `code`, `document`, `paper`, `image`, `rationale`, `concept`
  - `source_file`: relative path to the source
  - `source_location`: line number (e.g. `L207`)
  - `community`: integer community ID from the Leiden algorithm
- **Edge** = a typed relationship. Relations include:
  - `imports_from` (EXTRACTED, confidence 1.0)
  - `calls` (EXTRACTED)
  - `references` (EXTRACTED)
  - `cites` (EXTRACTED, doc-to-doc)
  - `conceptually_related_to` (INFERRED)
  - `semantically_similar_to` (INFERRED)
  - `rationale_for` (INFERRED, design-decision edges)
- **Community** = a cluster of nodes Leiden found to be densely
  interconnected. 83 communities at v1.0.0, 47 of which have human-
  chosen labels in `GRAPH_REPORT.md`.

### When to use graphify

**Use `graphify-out/graph.html`** to:
- Find where a feature lives (visual click-and-explore)
- Trace a data flow end-to-end (open the html, find the start node,
  click around)
- Verify a new feature integrates with the right modules

**Use `graphify-out/GRAPH_REPORT.md`** to:
- Get the list of god nodes (most connected symbols)
- See surprising cross-module connections
- Get suggested questions to ask the graph
- Audit the codebase structurally

**Use `graphify-out/graph.json`** programmatically (the file is
NetworkX node-link JSON, loadable from Python or any JSON parser).

**Use the `graphify` CLI** to:
- `graphify update .` вЂ” incremental rebuild after code changes
- `graphify query "<question>"` вЂ” BFS or DFS traversal by concept
- `graphify path "SymbolA" "SymbolB"` вЂ” shortest path
- `graphify explain "Symbol"` вЂ” neighbors + descriptions
- `graphify export html` вЂ” regenerate graph.html

### When NOT to use graphify

- **For test counts, file locations, code reading.** `grep` / `cargo
  test` are faster.
- **As the design rationale.** Section 1.
- **To track project state.** Section 1's STATE.md and ROADMAP.md.

## 4. v1.1 goal (what we are building)

**v1.1.0 = a live-trading binary that submits Jito bundles on Solana
mainnet, with a daily SOL loss cap, in milliseconds from detection
to submission.**

Concretely:

1. **Detection в†’ sign в†’ submit** end-to-end latency < 80 ms (the
   project-defined budget for `t_detect_ms + t_decide_ms` from
   `LatencyBudget`).
2. **Bundle hits Jito tip router** as a single 5-tx bundle.
3. **Daily SOL loss cap** enforced in `dl-signer`. If the cap is hit,
   the signer refuses new bundle signatures until the next UTC day.
4. **A `dl-app run` subcommand** is the live binary. The `dl-app run
   --dry-run-live` flag replays a capture and *would have submitted*
   the bundles, but writes the would-have-submitted bundles to a
   log instead of Jito. This is the validation gate.
5. **Submission metrics** emit via the existing `dl-app metrics prom`
   endpoint with new counters:
   - `bundles_submitted_total`
   - `bundles_landed_total`
   - `bundles_failed_total{reason=...}`
   - `sol_spent_total`
   - `sol_received_total`
   - `daily_cap_remaining_sol`
   - `last_submission_latency_ms`

## 5. Decisions locked (4 architectural choices)

These four choices were decided by the user before this plan was
written. They cascade through the entire v1.1 design.

### Decision 1: Hot wallet with daily SOL cap

**What it is:** A private key stored in an encrypted file on the
host machine. `dl-signer` reads it at boot, holds it in process
memory, and refuses to sign a bundle if doing so would breach the
daily cap.

**Why it was chosen:** Fastest to build (~1 day), simplest operational
model, matches the project's single-host single-operator scope.

**Risk acknowledged:** A single host compromise drains the daily
cap. Mitigated by:
- Encrypted keyfile (`aes-256-gcm` with key derived from
  `DL_SIGNER_PASSPHRASE` env var or `argon2id` from a file)
- Hard cap of 5 SOL/day (default; configurable via `DL_DAILY_CAP_LAMPORTS`)
- Per-bundle cap of 0.5 SOL (default; `DL_PER_BUNDLE_CAP_LAMPORTS`)
- Rate limit: max 10 bundles/minute (`DL_BUNDLES_PER_MINUTE`)
- Process exit on suspicious activity (more than 3 failed signatures
  in 60 seconds, or any signature to a non-whitelisted program)

**Where it lives:** `crates/dl-signer/` (new crate, Section 6).

### Decision 2: Jito bundles only

**What it is:** The executor submits up to 5 transactions as a Jito
bundle via `jito-bundle::send_bundle`. No fallback to direct RPC
submission, no BAM, no spam-the-block strategy.

**Why it was chosen:** Single substrate, well-documented, the
battle-tested path today. BAM is future-work.

**Risk acknowledged:** If Jito's Block Engine is degraded, no trades.
Mitigated by:
- Health check on the Jito Block Engine at boot (fail fast)
- Bundle timeout: 30 seconds. If no landing confirmation by then,
  mark as `lost` and continue
- Operator can flip a config flag to disable execution entirely
  (back to paper mode) without restarting the binary

**Where it lives:** `crates/dl-executor/src/jito.rs` (Section 6).

### Decision 3: Jupiter Aggregator v6

**What it is:** The executor uses Jupiter's public quote + swap
endpoints to get the best route through Raydium / Orca / Meteora
swap instructions for a given input/output token pair. Jupiter
returns the swap-instruction bytes; the executor wraps them in a
Jito bundle with a tip transaction.

**Why it was chosen:** Covers all three AMM families in one call,
~2 days of integration, broadest coverage. Our hand-rolled fill
math in `dl-sim` stays as the simulator-side oracle вЂ” we use it
to decide *whether* to send the bundle, then Jupiter tells us
*how*.

**Risk acknowledged:** Jupiter API downtime or rate limit. Mitigated
by:
- 50 ms request timeout, 3 retries with exponential backoff
- Local quote cache: 100 ms TTL, populated from the most recent
  Jupiter response for the same input/output/mint
- Operator can pin Jupiter to a specific API endpoint via
  `DL_JUPITER_API_URL` (default: `https://quote-api.jup.ag/v6`)

**Where it lives:** `crates/dl-executor/src/jupiter.rs` (Section 6).

### Decision 4: Multi-plan series (3 sub-plans)

**What it is:** v1.1 is split into 3 shippable sub-plans:
- `08-01-PLAN.md` вЂ” executor + hot-wallet signer
- `08-02-PLAN.md` вЂ” streaming detector + latency benchmark

**`crates/dl-app/src/live.rs`** (new, ~300 LoC) вЂ” The glue between

```rust
pub async fn run_live_pipeline(cfg: LiveConfig) -> Result<(), LiveError> {
    let mut detector = StreamingGraph::new();
    let mut feed = open_feed(&cfg.feed_kind, &cfg.feed_path)?;
    let signer = dl_signer::boot(&cfg.keyfile, &cfg.passphrase, &cfg.cap_config)?;
    let jito = dl_executor::JitoClient::new(&cfg.jito_url)?;
    let jupiter = dl_executor::JupiterClient::new(&cfg.jupiter_url)?;
    let mut ledger = dl_ledger::LedgerWriter::new(File::create(&cfg.ledger_path)?)?;
    let mut metrics = MetricsRegistry::new();

    while let Some(ev) = feed.next_event() {
        if let FeedEvent::AccountUpdate { account, slot, .. } = ev {
            let pool = dl_state::decoder::decode_account(&account)?;
            if let Some(cycles) = detector.on_pool_update(pool) {
                for cycle in cycles {
                    let t0 = Instant::now();
                    let would_trade = ev_conservative_pnl(&cycle) > 0;
                    metrics.observe("t_detect_ms", t0.elapsed().as_millis() as f64);
                    if !would_trade { continue; }

                    let bundle = dl_executor::submit_opportunity(&cycle, &signer, &jito, &jupiter, &cfg.tip_config).await?;
                    metrics.observe("t_submit_ms", t0.elapsed().as_millis() as f64);

                    let entry = LedgerEntry::from_cycle(&cycle, bundle.tip_lamports);
                    ledger.write_entry(&entry)?;
                    metrics.counter("bundles_submitted", 1);
                }
            }
        }
    }
    Ok(())
}
```

### 9.2 Failure modes

| Failure | Detection | Recovery |
|---|---|---|
| Keyfile corrupted | `KeyStore::from_encrypted` returns `SignerError::BadCipher` | Operator recreates keyfile from seed phrase |
| Wrong passphrase | 3 strikes в†’ process exit | Operator restarts with correct passphrase |
| Daily cap breached | `CapState::try_charge` returns `CapExceeded` | Process refuses new signatures; operator drains wallet at UTC midnight |
| Jupiter API down | 50ms timeout Г— 3 retries в†’ `ExecutorError::JupiterDown` | Bundle is not built; opportunity is skipped |
| Jito Block Engine down | Health check at boot fails в†’ `JitoClient::new` returns error | Process refuses to start; operator waits for Jito to recover |
| Bundle timeout (30s, no landing) | `JitoClient::poll_landing` returns `LandingResult::Lost` | Bundle is logged as lost; opportunity is not retried |
| Signer process crash | `dl-app` detects EOF on the signer's stdout | Process exits with `RunExit::SignerCrashed`; operator restarts |
| Ledger write failure | `LedgerWriter::write_entry` returns `Io` error | Process exits with `RunExit::LedgerError`; operator inspects disk |
| Clock drift | `chrono::Utc::now()` and `Instant::now()` are monotonic; cap reset uses `chrono` | Operator syncs NTP |

## 10. Risks and honest caveats

### 10.1 Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Hot wallet drained via host compromise | Medium | 5 SOL/day | Encrypted keyfile, daily cap, monitor externally |
| Jupiter API rate-limits us | Medium | Missed opportunities | 100ms local quote cache, 3 retries with backoff |
| Jito Block Engine downtime | Low | No trades for the duration | Boot health check, fail fast, operator wait |
| Bundle signature too slow (>10ms) | Low | Lose the race | Benchmark in 08-02, 0.5 SOL per-bundle cap as the worst-case |
| `dl-sim` per-cycle tip model wrong | High | Conservative bound is wrong | v1.0 06-02 SUMMARY already flags this; the v1.1.0-executor tag has a 7-day validation gate |
| Anchor dataset still synthetic | Already known | Calibration is provisional | v1.1 ships anyway; v1.2 swaps in live Jito API pull |
| Orca multi-tick fill error | Already known | ~1-2% EV error near threshold | v1.1 ships anyway; v1.2 multi-tick fix |
| Borsh/Anchor version skew in Jupiter swap | Low | Deserialization fails | Use Jupiter's official `jupiter-aggregator-client` Python model for the swap-tx shape, port to Rust with tests |
| Regulatory: MEV bots are not illegal in most jurisdictions but a future change could complicate things | Low | Shutdown | Open-source AGPL/Apache license, the codebase is auditable |

### 10.2 Honest caveats (carried from v1.0 + new)

1. **Per-cycle tip in `dl-sim`** вЂ” added in 08-01, verified in 08-02.
2. **`calibrate()` is a heuristic** вЂ” same status; v1.2 follow-up.
3. **The Orca single-tick fill** вЂ” same status; v1.2 multi-tick fix.
4. **Live Jito API pull** вЂ” same status; v1.2 follow-up.
5. **DSR formula constants** вЂ” same status; v1.2 follow-up.
6. **Metrics emission sites in lower crates** вЂ” 08-02 wires them
   in as part of the streaming work.
7. **The hot-wallet key is on a single host** вЂ” a host compromise
   drains the daily cap. The 5 SOL/day cap is the explicit
   tradeoff for shipping fast.
8. **Jupiter API is a single point of failure** вЂ” the operator
   can pin to a specific endpoint (run-your-own Jupiter fork), but
   the default uses jup.ag.
9. **Bundle landing confirmation is best-effort** вЂ” Jito's
   `poll_landing` is poll-based, not push-based; we can lose
   land-or-no-land races in the 30-second timeout window.

### 10.3 What we are NOT doing in v1.1

- **Sandwiching** вЂ” permanently out of scope.
- **CEX-DEX arbitrage** вЂ” out of scope.
- **New-pool sniping** вЂ” out of scope.
- **JIT liquidity provision** вЂ” out of scope.
- **Multi-block / cross-block strategies** вЂ” v1.1 is single-block atomic only.
- **MEV on EVM chains** вЂ” Solana only.
- **BAM (Block Engine Marketplace)** вЂ” v1.2 follow-up.
- **HSM key custody** вЂ” v1.2 follow-up.
- **Sidecar signer** вЂ” v1.2 follow-up (decision matrix in Section 5.1).
- **Session keys with auto-expiry** вЂ” v1.2 follow-up.

## Appendix B: Glossary

| Term | Definition |
|---|---|
| AMM | Automated Market Maker (the DEX that holds the liquidity) |
| Atomic | All-or-nothing: a transaction either lands as a whole, or doesn't land at all |
| Bundle | A set of up to 5 transactions submitted to Jito that all land together or not at all |
| CFMM | Constant Function Market Maker (e.g. Raydium's x*y=k) |
| DLMM | Dynamic Liquidity Market Maker (Meteora's bin-based model) |
| Cycle | A path through the price graph that starts and ends at the same token, with a negative total weight (i.e. profitable) |
| EV | Expected Value: `E[PnL] = p Г— net_per_outcome` |
| HSM | Hardware Security Module: a physical device that holds keys and signs without exposing the key |
| Jito | A Solana validator fork that runs a Block Engine for MEV bundle auction |
| Lamport | The smallest unit of SOL. 1 SOL = 1,000,000,000 lamports |
| MEV | Maximal Extractable Value: profit a block producer (or someone with priority access to them) can extract by reordering/inserting transactions |
| p_land | Probability that a submitted bundle actually lands on-chain |
| p_win | Probability that we win the auction (vs other searchers bidding on the same opportunity) |
| p_detect | Probability that we detected the opportunity in time (i.e. before the price moved) |
| Pubkey | Solana public key (32 bytes, base58-encoded) |
| Q64.64 | Fixed-point representation: 64 integer bits + 64 fractional bits in a `u128` |
| Slippage | The difference between expected and actual fill price |
| Tip | A lamport transfer to a Jito validator to bid for bundle inclusion in the next block |
| Whirlpool | Orca's concentrated-liquidity AMM |
