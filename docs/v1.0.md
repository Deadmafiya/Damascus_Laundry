# damascus_laundry v1.0.0 — Release Notes

## What is this

`damascus_laundry_v1.0` is a Solana MEV **paper-trading** engine. It
ingests real-time Solana market state, detects **atomic
DEX-DEX arbitrage** opportunities across three AMM families
(Raydium AMM v4, Orca Whirlpool, Meteora DLMM), and *simulates*
whether each opportunity would have been profitable — **without
ever submitting a real transaction**.

**Paper trading only.** v1.0 holds no private keys, signs
nothing, and submits nothing. There is no funded wallet in the
value path. Live execution is explicitly out of scope for v1.0.

## What ships in v1.0

- **Multi-DEX detection**: Raydium constant-product + Orca
  Whirlpool (concentrated-liquidity, Q64.64) + Meteora DLMM
  (per-bin). Per-DEX edge labeling on the price graph lets
  multi-DEX triangles enumerate.
- **Conservative-by-default simulation**: every opportunity is
  evaluated with both an optimistic bound (no haircut) and a
  conservative bound (full failed-cost stack). The trade gate
  fires on the conservative bound only. This is the structural
  answer to "the simulator said yes, the live trade would have
  lost".
- **Gold-file replay + DST**: `dl-recon` re-runs the full
  pipeline against a captured window and asserts byte-identical
  output. 5 fault-injection middlewares (`BoundedDrop`,
  `BoundedCorrupt`, `JitteredSlot`, `Reorder`, `BoundedDrop` +
  `JitterRng`) cover the realistic failure modes. 6 invariants
  pin the determinism / no-silent-lies contract.
- **Paper ledger (v3)**: every evaluated cycle is recorded in a
  `DLD-LDG1` v3 ledger. `LedgerSummary` exposes total /
  would_trade counts, sum_optimistic / sum_conservative e_pnl,
  median and p95 percentiles.
- **Deflated Sharpe / purged walk-forward CV / PBO**: the
  `dl-recon-overfit` crate prevents the calibrated
  `EvalParams` from overfitting. (The PDF cross-check on the
  DSR formula constants remains an open gate; the formula is
  range-tested, not value-pinned.)
- **Configurable engine**: `EngineConfig` (TOML) + 20+
  `DL_*` env vars tune the engine without recompile. `dl-app
  config print` dumps the active config.
- **Prometheus metrics**: `dl-app metrics prom [--port N]`
  exposes `/metrics` in Prometheus text format. Hand-rolled
  text-format emitter (no `prometheus` crate dep) to keep the
  workspace integer-only.
- **Reproducible paper PnL reports**: `scripts/
  reproduce_paper_pnl.sh --capture <fixture> --out <dir>`
  produces a v3 ledger + recon report JSON + Markdown
  summary, deterministically, from a single command.

## What does NOT ship in v1.0

- **Live trade execution**. v1.0 is paper-trading. A future
  phase will swap in only the executor module; the value
  path is unchanged.
- **BAM-era tip routing**. Open question in
  `.paul/research/onchain-arb-anchor-dataset.md` §1.6.
- **Sandwiching**. Project policy: out of scope.
- **Other DEXes** (Phoenix, OpenBook).
- **Custom dashboard UI**. A Grafana import of the Prometheus
  output is the v1.0 path.
- **Real on-chain anchor dataset for calibration**. The
  `anchors.v0.jsonl` shipped with v1.0 has placeholder
  numbers sourced from `jito-labs/mev-bot` constants and the
  Helius MEV report; replacing it with a live Jito API pull
  is a v1.0+ follow-up.
- **PDF cross-check on the Deflated Sharpe formula
  constants**. The formula is implemented and range-tested
  but the exact reference values from the Bailey & López
  de Prado 2014 paper are not pinned to test vectors.

## How to reproduce a paper PnL report

```bash
# 1. Build the engine.
cargo build --release -p dl-app

# 2. Run a one-shot reproduction. Requires a capture file
#    (a `.bincode` artifact of `dl-app`'s live-capture path,
#    or any of the test fixtures in `crates/dl-feed/tests/fixtures/`).
./scripts/reproduce_paper_pnl.sh \
    --capture crates/dl-feed/tests/fixtures/sample_capture.bincode \
    --out ./out

# 3. Inspect the three outputs:
ls -la ./out/
#   ledger.dld              DLD-LDG1 v3 paper ledger
#   report.json             ReconReport, JSON-serialized
#   PAPER_PNL_REPORT.md     Human-readable summary
```

The script refuses to run without `--capture` (no surprise
network calls). A `--live-capture` mode is documented in
the script's help text but is not part of the v1.0 contract;
live capture is a separate workflow.

## Live observability

```bash
# Start a Prometheus scrape endpoint on port 9090.
cargo run --release -p dl-app -- metrics prom --port 9090 &
curl http://127.0.0.1:9090/metrics
```

The endpoint serves the engine's `MetricsSink` output in
Prometheus text format. Five metrics land by default:
`opps_per_sec` (counter), `active_pools` (gauge),
`would_trade` (gauge), `total_tip_lamports` (gauge), and any
histograms the engine emits per-update. Threading the
registry through `dl-feed` / `dl-detect` / `dl-sim` /
`dl-recon` is a v1.0+ follow-up; for v1.0, the `dl-app`
layer is the only emitter.

## Configuration

The engine reads configuration from a TOML file (default
`config.toml` in the working directory) and `DL_*` env vars.
The env vars override the TOML values; both override the
hard-coded `EvalParams::conservative_default()` baseline.
`dl-app config print` dumps the active config to stdout.

```bash
# Override individual parameters:
DL_BASE_WIN_PPM=200000 DL_P_LAND_DECAY_PPM_PER_MS=200 \
  cargo run --release -p dl-app --bin dl-app -- config print
```

See `crates/dl-app/src/config.rs` for the full list.

## Honest caveats

1. **Per-cycle tip in `dl-sim`** is not yet implemented. The
   `ReconReport::total_tip_lamports` field sums the
   `LedgerEntry::tip_lamports` values, which are 0 by default
   for the v1.0 ledger. The harness reports 0 faithfully;
   the underlying simulation does not model tip yet. v1.0+
   work.
2. **`calibrate()` is a heuristic**, not a numerical
   optimizer. The fit walks `base_win_ppm` by the average
   signed divergence and re-checks tolerances with a
   fixed-point scale factor. Production would call the
   evaluator with the new params and re-derive the
   aggregates. v1.0+ work.
3. **Live Jito API pull** for on-chain anchor calibration is
   blocked on outbound network access from this host. The
   `anchors.v0.jsonl` shipped with v1.0 has placeholder
   numbers from `jito-labs/mev-bot` constants and the Helius
   MEV report; a live pull is a v1.0+ follow-up.
4. **The Orca Whirlpool fill math** is a single-tick
   approximation. Inputs that cross a tick boundary have a
   ~1-2% error vs. the full SDK reference. Multi-tick
   support is a v1.0+ follow-up.
5. **Metrics emission sites** in the lower crates (dl-feed /
   dl-detect / dl-sim / dl-recon) are not yet wired; the
   `MetricsSink` trait + `MetricsRegistry` are ready, and
   `dl-app` is the only emitter for v1.0.
